# 아보하 (Avoha) — MVP 기획서 & PRD v1.0

> "아무 일 없는 보통의 하루"를 채집·세공하는 게임화 아카이빙 서비스.
> 5일 Wizard-of-Oz 방식으로 핵심 루프를 유저 테스트.

---

## 0. Context — 이 문서가 쓰이는 이유

갓생·성과 중심 사회에 지친 20대 후반 여성의 **"평범한 하루에 대한 감수성"** 을 회복시키는 것이 제품 가설. 기존 `1_mvp/` 소확행 웹앱은 지도 기반 AI 자동 분석 플로우로 **투자자 데모 목적**이며 이번 유저 테스트와 병행 존속한다. 본 문서는 **별도 트랙**인 아보하(신규 제품) MVP의 기획·PRD·엔지니어링 상세를 정의한다.

**해결하려는 문제**: 일기/기록 앱은 또 다른 기록 압박이 되어 이탈률이 높다. 유저가 "기록한다"는 부담 없이 일상을 "수집/세공"하는 게임 경험을 제공하면 Day-3 잔존이 올라갈 것인가?

**원하는 결과 (MVP 성공 가설)**:
- **KPI#1 습관화**: 최초 유입자의 Day-3 잔존율 ≥ 50% (3일 차에도 1회 이상 채집 시도)
- **KPI#2 보상 전이**: 채집 완료 → 앱 진입 전환율 ≥ 60%
- **KPI#3 정성**: FGI(심층 인터뷰)에서 "무탈함에 대한 태도의 긍정 변화" 확인
- **부산물**: WoZ 운영 기간 동안 축적된 감정 분류 라벨로 포스트-MVP 자동 AI의 **프롬프트 정답지** 구축

**핵심 차별화**: 기록은 카카오톡(자연 채널), 보상은 픽셀 아트 게임 인벤토리. 두 공간이 분리되어 **"기록 피로"** 를 피한다. AI 선-제안 + 운영자 확정의 하이브리드로 정성 품질을 유지하면서 동시에 자동화 학습 데이터를 얻는다.

### 3.4 실시간 동기화 패턴
카톡에서 채집이 완료되면 앱 내 홈 필드가 **자동 반짝임**. 구현:
- 백엔드 `/sse/inventory?user_id=xxx` (Server-Sent Events) 또는 PostgreSQL `LISTEN/NOTIFY` 경유
- 배터리 보호: 백그라운드 시 polling 60s로 폴백
- `useRealtimeInventory` 훅이 추상화, 새 아이템 도착 시 `field-store`에 push

### 3.5 PWA·성능 예산
- LCP < 2.5s, TTI < 3.5s (3G Slow 기준)
- 번들: initial < 180 KB gzip (sprite atlas는 lazy)
- Lighthouse 접근성 ≥ 90 (픽셀 이미지 alt 필수)
- 오프라인 인벤토리 캐시 (최근 24시간)

## 4. 백엔드 파트 (Backend)

### 4.1 인프라 (Railway)
Railway 단일 프로젝트 하위 6개 서비스:

| 서비스 | 스펙 | 용도 |
|---|---|---|
| `avoha-api` | 1 vCPU / 1 GB | Fastify HTTP API, OAuth, 비즈 로직 |
| `avoha-agent` | 0.5 vCPU / 512 MB | AI 분류·리액션 생성 워커 (Redis 큐 consume) |
| `avoha-rembg` | 1 vCPU / 1 GB | 누끼 컨테이너 (rembg python) |
| `postgres` | managed | 관계형 저장소 |
| `redis` | managed | 작업 큐 + SSE pub/sub |
| `metabase` | 1 vCPU / 2 GB | 분석 대시보드 |

- Railway Volume 10 GB: 원본 이미지 + 누끼 결과물
- 도메인: `api.avoha.today` (확정 필요), `avoha.today` (프론트), `console.avoha.today` (운영)
- TLS: Railway 자동 (Let's Encrypt)
- 비용 실측 예산: **$30–40 / 5일** (Pro 플랜 $20 포함 시 추가 $10–20)

### 4.2 스택
- **Node.js 22 + TypeScript 5.7 + Fastify 5**
- **Drizzle ORM** (Zod 스키마 연동)
- **Pino** 구조화 로깅 → `stdout` → Railway 로그
- **BullMQ** (Redis 기반 큐) — AI 분류·누끼 비동기
- **Kakao Biz SDK**: 공식 상담톡 API + 알림톡 (주문형 알림톡 템플릿 선-승인)

### 4.3 데이터 모델 (Postgres, Drizzle)

```sql
-- 사용자
CREATE TABLE users (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  kakao_id        BIGINT UNIQUE NOT NULL,
  nickname        TEXT NOT NULL,
  profile_url     TEXT,
  joined_at       TIMESTAMPTZ DEFAULT now(),
  consent_version TEXT NOT NULL,           -- 동의서 버전
  deleted_at      TIMESTAMPTZ
);

-- 일일 채집권
CREATE TABLE collection_tickets (
  user_id         UUID REFERENCES users(id),
  date            DATE,
  remaining       SMALLINT NOT NULL DEFAULT 5,
  last_refilled_at TIMESTAMPTZ,
  PRIMARY KEY (user_id, date)
);

-- 카톡 수신 메시지 (raw)
CREATE TABLE kakao_messages (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID REFERENCES users(id),
  received_at     TIMESTAMPTZ DEFAULT now(),
  content_type    TEXT CHECK (content_type IN ('text', 'image')),
  body            TEXT,                      -- text
  media_url       TEXT,                      -- image
  status          TEXT DEFAULT 'pending',    -- pending|proposed|confirmed|rejected
  ai_suggestion   JSONB,                     -- { emotion, reaction, confidence }
  operator_id     UUID,
  finalized_at    TIMESTAMPTZ
);

-- 감정 분류 카탈로그
CREATE TABLE emotions (
  code            TEXT PRIMARY KEY,          -- 'satiation', 'fortune', ...
  name_ko         TEXT NOT NULL,             -- '포만감'
  gem_name        TEXT NOT NULL,             -- '황옥'
  hex_color       TEXT NOT NULL,
  trigger_keywords TEXT[]
);

-- 광물 (감정 원석)
CREATE TABLE gems (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID REFERENCES users(id),
  emotion_code    TEXT REFERENCES emotions(code),
  tier            SMALLINT NOT NULL,         -- 1..4
  source_message_id UUID REFERENCES kakao_messages(id),
  crafted_from    UUID[],                    -- tier>1일 때 재료 gem id들
  created_at      TIMESTAMPTZ DEFAULT now(),
  consumed_at     TIMESTAMPTZ                -- 세공 재료로 소모된 시각
);

-- 스티커 (누끼 이미지)
CREATE TABLE stickers (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID REFERENCES users(id),
  source_message_id UUID REFERENCES kakao_messages(id),
  image_url       TEXT NOT NULL,             -- 누끼 후 결과
  polaroid_fallback BOOLEAN DEFAULT FALSE,   -- 누끼 실패 시 프레임
  placed_on_field BOOLEAN DEFAULT FALSE,
  created_at      TIMESTAMPTZ DEFAULT now()
);

-- 레시피 (이종 합성 카탈로그)
CREATE TABLE recipes (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug            TEXT UNIQUE,               -- 'dinner_star'
  name_ko         TEXT,                      -- '저녁 식탁의 별'
  ingredient_codes TEXT[] NOT NULL,          -- ['satiation', 'warmth']
  result_tier     SMALLINT NOT NULL,
  unlocked_by     TEXT
);

-- 세공 이벤트
CREATE TABLE crafting_events (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID,
  ingredient_ids  UUID[],
  result_id       UUID REFERENCES gems(id),
  recipe_slug     TEXT,                      -- 이종 합성 시
  created_at      TIMESTAMPTZ DEFAULT now()
);

-- 범용 이벤트 로그 (KPI)
CREATE TABLE events (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID,
  event_type      TEXT NOT NULL,             -- 'collect', 'app_enter', 'craft', ...
  props           JSONB,
  occurred_at     TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX ON events (event_type, occurred_at);
CREATE INDEX ON events USING gin (props);
```

### 4.4 API 계약 (요약)

```
GET  /auth/kakao/callback            → 세션 발급
GET  /me                              → 프로필 + 채집권 잔여
GET  /inventory/gems                  → 광물 (필터: emotion, tier)
GET  /inventory/stickers              → 스티커
POST /crafting/combine                { ingredient_ids: uuid[] } → gem
GET  /crafting/recipes                → 해금된 레시피 카드
GET  /field/today                     → 오늘 드롭된 아이템 좌표
GET  /sse/inventory                   → 실시간 업데이트 스트림
POST /events                          → 클라이언트 이벤트 로깅

# 운영 콘솔
POST /webhook/kakao                   ← Kakao Biz 수신
GET  /ops/queue                       → 대기 메시지 (AI 제안 포함)
POST /ops/messages/:id/confirm        { emotion_code, reaction_text } → 확정
POST /ops/messages/:id/reject         → 사용자에 재질문 송신
GET  /ops/dashboard-metrics           → 실시간 지표
```

### 4.5 카카오톡 웹훅 플로우

1. 유저 → 채널 메시지 송신
2. Kakao Biz → `POST /webhook/kakao`
3. 백엔드: `kakao_messages` insert, Redis `emotion-queue`에 push
4. `avoha-agent`: 큐 consume → GPT-4.1 mini/Gemini 2.5 Flash로 감정 후보 + 리액션 초안 생성 → `ai_suggestion` 업데이트, 상태 `proposed`
5. 운영 콘솔 SSE로 새 메시지 알림 팝
6. 운영자 확인/수정/1-click 승인 → `status=confirmed`
7. `emotion_code` 기반 `gems` insert (tier=1), 채집권 -1
8. Kakao 메시지 발송: "[감정] 원석 지급! 남은 채집권 n장. [인벤토리 가기]"
9. 클라 SSE로 홈 필드 새 드롭 렌더

## 5. AI & 에이전트 파트 (AI)

### 5.1 역할 정의
WoZ 기간 동안 AI는 **운영자 보조** 역할. 자동 응답 아님. 학습 데이터 축적과 운영자 속도 향상에 집중.

### 5.2 에이전트 구성

| 에이전트 | 모델 | 입력 | 출력 | 호출 시점 |
|---|---|---|---|---|
| `emotion-classifier` | GPT-4.1 mini (주) / Gemini 2.5 Flash (폴백) | 유저 메시지 텍스트 | `{ top3_emotion_codes, confidence[], rationale }` | 수신 즉시 |
| `reaction-drafter` | GPT-4.1 mini | 유저 메시지 + 확정 감정 | 120자 이내 리액션 후보 2개 | 운영자 감정 확정 후 |
| `edge-guard` | 규칙 + GPT-4.1 mini | 유저 메시지 | `{ is_crisis, is_offensive, needs_human }` | 병렬 호출 |
| `sticker-tagger` | Gemini 2.5 Flash (Vision) | 원본 이미지 | `{ object, mood_tag, suggested_caption }` | 이미지 수신 시 |

### 5.3 프롬프트 관리
- 프롬프트는 `agent/prompts/*.md` 버전 관리
- 10종 감정 카탈로그를 시스템 프롬프트에 주입 (JSON 블록)
- Few-shot 10개: PM이 작성, 피봇 컨셉에 맞춘 실제 예시
- 응답 포맷 Zod 검증 → 실패 시 1회 재시도, 2회 실패 시 `needs_human` 처리

### 5.4 Edge-guard (안전 장치)
- **위기 신호** (자해·자살 언급): 즉시 인간 개입 + 상담 번호(109) 안내 카드
- **성희롱·어그로**: 운영자 확인 전 차단, 관리자 에스컬레이션
- **너무 짧은 입력** (`ㅇㅇ`, `ㅋㅋ`): "조금 더 설명해 주세요" 자동 리턴, 채집권 소모 없음

### 5.5 학습 데이터 파이프라인 (포스트 MVP 준비)
- `ai_suggestion` vs 운영자 최종 `emotion_code` 차이를 `events`에 기록
- 5일 후 CSV export: `message_text, ai_top1, ai_confidence, operator_final, match`
- 차이나는 케이스만 추려 → 프롬프트 Few-shot 보강 / fine-tune 데이터셋 초안

### 5.6 누끼 서비스 (`avoha-rembg`)
- Python + FastAPI + `rembg` (u2net_lite 모델, CPU, ~300 MB)
- 엔드포인트: `POST /remove-bg` (multipart)
- 결과를 Railway Volume에 저장, signed URL 반환
- 처리시간 SLA: p95 ≤ 8초
- 실패(얼굴 인식 안됨, 너무 복잡): 폴라로이드 프레임 폴백 (자동)

### 5.7 AI 파트 작업 리스트
- [AI-1] 프롬프트 v0 작성 (감정 분류·리액션·edge)
- [AI-2] OpenAI·Gemini key 환경 분리 + 비용 alarm ($30 한도)
- [AI-3] Few-shot 10셋 × 감정 × 스타일 템플릿
- [AI-4] 에이전트 워커 Fastify adapter
- [AI-5] Zod 검증 + 재시도 체인
- [AI-6] rembg 컨테이너 build & 배포
- [AI-7] 이미지 태거 (Gemini Vision)
- [AI-8] Edge-guard 테스트 케이스 30건
- [AI-9] 학습 데이터 export 스크립트
- [AI-10] 프롬프트 정답지 회고 문서 자동 생성

## 6. 디자인 파트 (Design)

### 6.1 비주얼 컨셉
**"레트로 픽셀 RPG × 따뜻한 디지털 저녁"**. 16-bit 감성이지만 고대비 피로 없이, 모던 일러스트 바디 + 픽셀 액센트. 아바타와 원석만 픽셀, 배경 UI는 부드러운 플랫.

### 6.2 컬러 시스템 (소확행 팔레트 계승)

| 토큰 | HEX | 용도 |
|---|---|---|
| `--coral` | `#E8614D` | 포인트 액션(세공 버튼) |
| `--mint` | `#3AAFA9` | 평온·여유 감정 & CTA 보조 |
| `--amber` | `#E8A838` | 채집권·포만감 |
| `--pixel-bg-dawn` | `#F4EBD9` | 홈 필드 08:00 레이어 |
| `--pixel-bg-dusk` | `#3A3E5B` | 홈 필드 18:00 레이어 |
| `--ink` | `#1B1C26` | 본문 텍스트 |
| `--parchment` | `#FBF7EE` | 카드/도감 배경 |

시간대별 배경 자동 전환 (AM/PM 파샤).

### 6.3 타이포 시스템
- 본문: Noto Sans KR 400/700
- 제목·도감: Noto Serif KR 700 + 픽셀 윤곽 stroke
- 숫자·지수: DungGeunMo (픽셀 한글체, 상업 이용 OK)

### 6.4 픽셀 에셋 구성

| 범주 | 출처 | 작업량 |

### 6.5 화면 와이어프레임 요약

- **W-01 카톡 채널**: 하단 고정 메뉴 `[오늘 채집 | 인벤토리 열기 | 도감 | 도움말]`. 운영자 응답에는 퀵 버튼 `[맞아요] [조금 달라요]`.
- **W-02 홈 필드**: 3-레이어 패럴랙스 (원경 구름, 중경 숲, 전경 풀밭). 아바타 idle 3초마다 깜빡. 오늘 채집 원석은 좌표(`field_drop_positions`)에 glow + 탭하면 "언제·뭐라 적어서 얻은 건지" 툴팁.
- **W-03 인벤토리**: 탭 [광물] [스티커]. 4×N 그리드, 등급 테두리 4색. 롱프레스 시 히스토리 바텀시트.
- **W-04 세공소**: 중앙 모루(픽셀), 슬롯 2개. 드래그앤드롭 피드백, `[세공 시작]` 망치 버튼. 성공 시 8프레임 파티클 → 상위 보석 등장 → 햅틱(웹 vibrate API).
- **W-05 도감**: 그리드 셀 40개(광물)+ 10개(레시피) + 스티커 무한. 미획득은 실루엣, 획득 시 뒤집힘 애니메이션.
- **W-06 마이페이지**: 상단 아보하 지수 그래프(최근 14일), 연속일수, 카카오 연동 상태, 데이터 내려받기·탈퇴.

### 6.6 인터랙션·모션 원칙
- 기본 이징: `cubic-bezier(0.22, 1, 0.36, 1)` (out-Quint, 부드럽고 탄력)
- 세공 성공 = 500ms 파티클 + 200ms 플래시 + 1s 사운드 (옵션)
- 홈 필드 새 아이템 드롭 = 위에서 떨어지는 애니메이션 + 바닥 먼지 이펙트
- 80ms 이하 마이크로 햅틱: 세공 완료·채집 확정

### 6.7 접근성
- 색상 외 형태로 감정 구분 (원석 모양 10종 고유화)
- 스크린리더용 `alt` 전수 작성
- 모션 감소 요청(`prefers-reduced-motion`)시 파티클 생략

**이종 합성 (레시피 카드 필요)**

| `destined_spark` | 운명의 조각 | 행운 + 설렘 | Lv.3 |
| `flow_crystal` | 흐르는 시간의 정수 | 여유 + 몰입 | Lv.4 |
| `homecoming_opal` | 돌아온 저녁의 오팔 | 안도 + 평온 | Lv.3 |
| `quiet_ember` | 조용한 불씨 | 감사 + 평온 | Lv.3 |

나머지 4종 레시피는 운영 기간 데이터 기반 추후 공개.

### 8.3 아보하 지수 산식
```
daily_index  = collect_count × 10
             + unique_emotions × 5
             + successful_crafts × 15
             + streak_days × 3
rolling_7d   = mean(daily_index[-7:])
normalized   = min(100, rolling_7d × 100 / 150)   # 150 = 이상적 상한
```

### 8.4 프라이버시·동의
- 카톡 채널 추가 시 봇 자동 메시지로 3줄 요약 고지 + 상세 약관 링크
- 동의 버전 `v2026.04`: `kakao_id`, 메시지 내용·이미지, 감정 라벨, 앱 이벤트 수집·저장 동의
- **운영자가 메시지 내용을 읽는다**는 점 명시 (WoZ 공개 투명성)
- 탈퇴·채널 차단 시: 30일 유예 → `users.deleted_at` set → 30일 후 개인식별 필드 파기
- 데이터 내려받기: `/me/export` (JSON + 이미지 zip)
